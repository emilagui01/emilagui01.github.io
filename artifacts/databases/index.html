Database Artifact

Overview

Travlr is a full-stack travel booking web application developed using the MEAN stack: MongoDB, Express, Angular, and Node.js. For the database category, this artifact focuses on the MongoDB and Mongoose portions of the application, including schemas, CRUD operations, indexed fields, validation, and database-driven application behavior.

This artifact was originally developed in CS-465 Full Stack Development I and is being used for the Database enhancement category of the CS-499 Computer Science Capstone.

Database Features

  • MongoDB database integration
  • Mongoose schema modeling
  • CRUD operations for trip management
  • RESTful API communication with database-backed routes
  • JWT-protected API endpoints
  • Indexed database fields
  • Validation for structured trip data

Project Structure

  • app_api/models/ - Mongoose schema definitions
  • app_api/controllers/ - Backend logic for database operations
  • app_api/routes/ - API endpoints connected to MongoDB operations
  • data/ - Seed and supporting data files
  • app_admin/ - Angular frontend that communicates with database-backed API routes

Running the Project

This artifact can be tested through the Travlr backend API, MongoDB database, Angular admin application, and Postman. Unlike the Software Design enhancement, the Database enhancement requires Postman and MongoDB because the goal is to verify backend validation, database behavior, protected API routes, and stored data directly.

Applications and Tools Needed

  • Visual Studio Code - Used to open and review the project files.
  • Node.js and npm - Used to install dependencies and run the Express backend server.
  • Angular CLI - Used to run the Angular administrator application when testing from the UI.
  • MongoDB / MongoDB Compass - Used to run the local database and verify stored trip and user records.
  • Postman - Used to test API requests directly, including valid data, invalid data, JWT-protected routes, and duplicate user registration.
  • Web Browser - Used to view the Angular admin interface at http://localhost:4200.

Since generated dependency folders such as node_modules were removed to reduce file size, dependencies must be reinstalled before running the project.

Start MongoDB locally or confirm the MongoDB service is running.

Install backend dependencies from the main Travlr project folder:

npm install

Install Angular dependencies:

cd app_admin
npm install

Start the backend application from the main Travlr project folder:

npm start

Start the Angular development server in a separate terminal:

cd app_admin
ng serve

Open the Angular admin application in a browser: 

http://localhost:4200

Open Postman to test backend API routes directly: 

http://localhost:3000/api/register
http://localhost:3000/api/login
http://localhost:3000/api/trips
http://localhost:3000/api/trips/:tripCode

Open MongoDB Compass to confirm that valid records are saved and invalid records are rejected.

Testing and Validation

The following tests were used to verify the Database enhancement for the Travlr application. These tests confirm that MongoDB, Mongoose validation, controller error handling, user validation, and JWT-protected database access work correctly.

1. Start the Required Services

  1. Start MongoDB locally or confirm that the MongoDB service is running.
  2. Start the backend server with npm start.
  3. Start the Angular admin application with cd app_admin and ng serve.
  4. Open Postman for API testing.

Expected Result: MongoDB, the Express backend, Angular, and Postman should all be ready for testing.

2. Valid User Registration Test

  1. In Postman, create a POST request to http://localhost:3000/api/register.
  2. Use the following JSON body:
{
  "name": "Database User",
  "email": "databaseuser@test.com",
  "password": "password123"
}

Expected Result: The API should return 200 OK with a JWT token.

3. Duplicate Email Registration Test

  1. Send the same registration request again using the same email address.

Expected Result: The API should return 409 Conflict with a message stating that a user with this email already exists.

4. Invalid Email Registration Test

  1. Send a POST request to http://localhost:3000/api/register.
  2. Use the following JSON body:
{
  "name": "Database User",
  "email": "notanemail",
  "password": "password123"
}

Expected Result: The API should return 400 Bad Request with a user validation error.

5. Login and JWT Token Test

  1. In Postman, create a POST request to http://localhost:3000/api/login.
  2. Use valid login credentials.
  3. Copy the returned JWT token.

Expected Result: The API should return a valid JWT token that can be used for protected trip requests.

6. Valid Trip Creation Test

  1. Create a POST request to http://localhost:3000/api/trips.
  2. In the Authorization tab, choose Bearer Token and paste the JWT token.
  3. Use the following JSON body:
{
  "code": "DBTEST2026",
  "name": "Database Test Trip",
  "length": "4 nights",
  "start": "2026-05-07",
  "resort": "Test Resort",
  "perPerson": 799.99,
  "image": "reef1.jpg",
  "description": "This is a valid database test trip."
}

Expected Result: The API should return 201 Created, and the trip should appear in MongoDB.

7. Invalid Trip Creation Test

  1. Send a POST request to http://localhost:3000/api/trips using the same Bearer token.
  2. Use invalid trip data:
{
  "code": "ab",
  "name": "x",
  "length": "y",
  "start": "",
  "resort": "z",
  "perPerson": "abc",
  "image": "reef1",
  "description": "short"
}

Expected Result: The API should return 400 Bad Request with a validation error, and the backend server should stay running.

8. Single Trip Lookup Test

  1. Send a GET request to http://localhost:3000/api/trips/DBTEST2026.

Expected Result: The API should return one trip object instead of an array because the backend uses findOne() for single-trip lookup.

9. Valid Trip Update Test

  1. Create a PUT request to http://localhost:3000/api/trips/DBTEST2026.
  2. Use the Bearer token for authorization.
  3. Use valid updated trip data:
{
  "code": "DBTEST2026",
  "name": "Database Test Trip Updated",
  "length": "5 nights",
  "start": "2026-05-07",
  "resort": "Updated Test Resort",
  "perPerson": 899.99,
  "image": "reef2.jpg",
  "description": "This is a valid updated database test trip."
}

Expected Result: The API should return 200 OK, and MongoDB should show the updated trip data.

10. Invalid Trip Update Test

  1. Send a PUT request to http://localhost:3000/api/trips/DBTEST2026.
  2. Use the Bearer token for authorization.
  3. Use invalid update data:
{
  "code": "DBTEST2026",
  "name": "x",
  "length": "y",
  "start": "",
  "resort": "z",
  "perPerson": "abc",
  "image": "reef1",
  "description": "short"
}

Expected Result: The API should return 400 Bad Request, and MongoDB should not overwrite the valid trip record.

11. Protected Route With No Token Test

  1. Send a PUT, POST, or DELETE request to a protected trip route.
  2. Set Authorization to No Auth.

Expected Result: The API should return 401 Unauthorized.

12. Protected Route With Bad Token Test

  1. Send a protected trip request.
  2. Set Authorization to Bearer Token.
  3. Use badtoken123 as the token value.

Expected Result: The API should return 401 Unauthorized.

13. Valid Delete Trip Test

  1. Create a test trip using the valid POST request.
  2. Send a DELETE request to http://localhost:3000/api/trips/DBTEST2026.
  3. Use the Bearer token for authorization.
  4. Refresh MongoDB Compass or the trip listing page.

Expected Result: The trip should be removed from MongoDB and no longer appear in the application.
